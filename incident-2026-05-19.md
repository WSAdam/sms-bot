# Incident Report — Firestore Read Quota Exceeded

**Date:** 2026-05-19
**First failure:** 09:59 AM ET (13:59 UTC)
**Service affected:** sms-bot (`/trigger/readymode` webhook + most read paths)
**Status:** Identified, fix in progress

---

## What happened

Around 10 AM ET on May 19, the sms-bot app started returning errors on its main webhook endpoint. The underlying cause was Firestore: we had exceeded our daily read quota, and Firestore began rejecting every read request from the app for the rest of the day.

## Why it happened

On May 5, a fix was deployed to the conversation-history lookup. The previous version had a bug where it would silently drop messages — for any phone whose records weren't in the first 500 documents Firestore returned, the lookup acted as if no history existed.

The fix raised the lookup limit from 500 to 50,000 so it could return every matching record. That solved the silent-data-loss bug, but it introduced a much more expensive problem: the lookup now reads **the entire conversations table** every time it runs, then filters the results in memory to find the few records that belong to the phone we actually asked about.

This lookup runs on every incoming dialer webhook — twice — once to check if the lead has opted out, and once to assemble the chat history for the AI prompt.

For two weeks the bug was effectively dormant. The conversations table was small, so each scan was cheap, and the total daily read volume stayed under our cap. But the table grows every day as new messages are saved. Every additional message in the table makes every future lookup slightly more expensive.

By May 19, the table had grown to roughly 10,000 records. With that table size, normal morning dialer traffic was enough to push our daily Firestore read count past the cap. Once it crossed, every subsequent read began failing with a quota error — which is what surfaced as the visible outage.

In plain terms: a small fix to one query turned a cheap lookup into one that scanned the whole table, and the cost has been quietly compounding ever since. May 19 was the day the math finally caught up with us.

## Why earlier days came close but didn't trip it

Firestore monitoring shows the same query has been spiking higher every day for two weeks. May 14 through May 18 all came uncomfortably close to the cap, but stayed just under. May 19's morning burst was the first one tall enough to push the daily total over the line. There was no special event yesterday — just one more day of growth on top of the previous day.

## Impact

- Inbound dialer webhooks (`/trigger/readymode`) began returning 500 errors at 09:59 AM ET and continued failing until the daily quota window reset.
- Rate-limiter and gates-config reads also failed during this window, so any path that touches Firestore was degraded.
- No data was lost. Failed requests were rejected before any write happened.
- Today (May 20) the app is operating normally, since the daily quota has reset.

## How we're fixing it

The root problem is straightforward: the lookup should ask Firestore "give me the messages for phone X" instead of "give me every message and I'll sort it out." Firestore can answer the first question cheaply with an index; the second question gets more expensive every day.

The fix has three parts:

1. **Stop scanning the whole table for per-phone lookups.** Change the conversation-history query to filter at the database layer using an indexed field, so each call only reads the handful of records that actually belong to that phone. This eliminates the per-request scan cost regardless of how large the table grows.

2. **Audit every other place we do the same thing.** Several other code paths use the same "read everything and filter in memory" pattern — the nightly conversation reseed, some dashboard endpoints, and an admin tool. Each one will be converted to a targeted query so the bug doesn't resurface from a different entry point.

3. **Add a safety rail.** Put a guard on full-table scans so any code that lists more than a small number of documents at once trips a warning in the logs. If a future change reintroduces this pattern by accident, we'll see it before it grows into another incident.

## Timeline to fix

- Same-day: deploy the per-phone lookup fix. This alone removes the dominant cost source.
- This week: audit and convert the remaining full-table scan call sites.
- This week: add the safety rail to surface any future regressions early.

## Lessons

- "Bigger limit" is not a fix for a query that returns the wrong subset of data. The right fix is a filter, not a larger ceiling.
- Costs that compound with table growth are invisible in testing and on quiet days. They show up later, suddenly, on a day that doesn't look special from the outside.
- Firestore monitoring already had the trend visible for two weeks before it broke. We weren't watching it. Going forward, that dashboard should be on the regular review list.
